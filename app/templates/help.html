1. Overview

ytdl-manager is a self-hosted YouTube subscription manager running in Docker on the MINI-S server (192.168.0.166, port 8911). It monitors YouTube channels and playlists on a schedule and automatically downloads new videos using yt-dlp.

  • Downloads are stored on the server's local disk, organized by channel name.
  • Each subscription has its own download archive — already-downloaded videos are never re-downloaded.
  • Video quality, check frequency, and filters are configurable per subscription.
  • The dashboard refreshes automatically every 5–10 seconds.

2. Using the Dashboard

Header

The header contains only the app title, yt-dlp version, Help link, and ⟳ Refresh button. The nav strip below the header links to the Dashboard, Job Status, and Container pages — the active page is highlighted in blue.

  • ⟳ Refresh — manually refreshes dashboard data. The timestamp next to it confirms when the last refresh ran.
  • yt-dlp [version] — shows the currently installed yt-dlp version. If a newer release is available on GitHub, an Update available button appears — click it to update without a full Docker rebuild.

Subscriptions panel header

  • Check All — immediately triggers a check for every subscription (asks for confirmation first).
  • + Add — opens the Add subscription page.

Stat bar

  • Subscriptions — total number of subscriptions configured.
  • Downloads Today — number of videos downloaded since midnight UTC.
  • Last Checked — which subscription was checked most recently and when.
  • Errors (today) — number of failed download runs today.

Subscription table

Click any row to open that subscription's log in the bottom panel. Use the filter pills (All / Errors / Downloading / Paused) to narrow the list. Each row has four action buttons:

  • Check — triggers an immediate check for that subscription only.
  • Edit — opens the Edit subscription page.
  • ⏸ / ▶ — pause or resume the subscription's scheduled checks.
  • Del — deletes the subscription after confirmation (downloaded files are kept).

Recent Downloads panel

Shows the 200 most recently downloaded videos, newest first. Updates automatically on each refresh cycle.

Log panel

Slides up from the bottom when you click a subscription row. Shows the full yt-dlp output log for that channel. Refreshes every 5 seconds while open. The Check Now button inside the panel triggers a new check for the currently viewed subscription.

3. Subscriptions

How subscriptions work

Each subscription monitors either a YouTube channel or a playlist. On each check cycle, ytdl-manager fetches the most recent video IDs and downloads any that haven't been seen before (tracked via a per-subscription archive file).

  • Channels — fetched via YouTube RSS feed; returns the 15 most recent videos.
  • Playlists — fetched using yt-dlp's flat playlist mode; returns all video IDs.

Check interval

Each subscription has an interval_hours setting. The scheduler checks whether enough time has elapsed since the last check before triggering a new one. Common intervals in use:

  • 1h — high-frequency channels (live cams, DVR channels)
  • 4–6h — regular content creators
  • 24h — infrequently-updated channels

Status values

  • ok (green) — last run completed successfully.
  • downloading (blue, spinning) — a check/download is actively running right now.
  • error (red) — last run ended with a non-zero exit code. Click the row to see the log.
  • idle (grey) — subscription is disabled or has never been checked.

4. Adding & Editing Subscriptions

Use the + Add button in the subscriptions panel header to open the Add page. Use the Edit button on any row to open the Edit page. Both pages have a Help link in the top-right that opens a full field reference in a new tab.

Fields

  • URL — YouTube channel (/@handle or /channel/UC...) or playlist (?list=...) URL. Pasting a /@handle URL auto-fills the Name and Output directory fields.
  • Name — display label used in the UI and logs. Editing the name also updates the output directory if it hasn't been manually changed.
  • Output directory — where videos are saved inside the container. Must start with /downloads/ — the prefix is added automatically if omitted.
  • Interval (hours) — how often the subscription is checked. After saving, use Stagger Schedules on the Job Status page to redistribute timing.
  • Quality — see Quality Settings section below.
  • date_after — playlists only; skip videos uploaded before this date. Accepts YYYYMMDD or relative values like today-30days.
  • Min duration (seconds) — skip videos shorter than this. Default 180 (3 minutes). Set to 0 to disable.
  • Exclude Shorts / Live / Past Live — filter checkboxes, all on by default.

5. Job Status Page

Accessible via the nav strip. Shows all recent and running download jobs and the upcoming schedule.

Jobs table

Click any row to load that subscription's log in the viewer panel on the right. The table auto-refreshes every 5 seconds when jobs are running, every 15 seconds otherwise. Use the filter pills (All / Running / Completed / Failed) to narrow the list.

  • Found — number of new video IDs discovered on this run.
  • Done — number successfully downloaded.
  • Fail — number that errored (non-zero exit).

Upcoming Scheduled Runs

Shows the next scheduled run time for every enabled subscription, sorted by time.

Stagger Schedules

If subscriptions start firing simultaneously, click Stagger Schedules to redistribute them evenly across their interval windows. Also run this after a container restart since APScheduler state is in-memory only.

6. Container Page

Accessible via the nav strip. Provides container status, cookie management, media scanning, and a log viewer.

Status card

Shows container name, running state, port mapping, API health, and current yt-dlp version. The Update yt-dlp button updates in place without a Docker rebuild. Restart and full rebuild must be done from the server shell.

Cookies card

Shows whether /data/cookies.txt is present, how many cookie lines it contains, and when it was last updated. Refresh Cookies (KBM) triggers the Keyboard Maestro macro on your Mac to run the automated cookie export flow. Verify Cookies does a quick sanity check on the file.

Media Library card

Scan Media triggers the Channels DVR scanner to pick up newly downloaded files.

Container Log

Shows recent download activity from /data/downloads.log. The pane fills the remaining viewport height. Use the line count selector and Reload button to control how much is shown.

7. Quality Settings

Each subscription has a quality setting that controls the yt-dlp format selector. Available options:

  • best — highest available resolution (no cap). May be 4K for channels that publish it.
  • 1440 — up to 1440p (2K). This is the default for all current subscriptions.
  • 1080 — up to 1080p Full HD.
  • 720 — up to 720p HD.
  • 480 — up to 480p SD.

All downloads prefer mp4 video with m4a audio and are merged into a single .mp4 file. If the preferred codec isn't available, yt-dlp falls back to the best available format at or below the target resolution.

💡 If videos look low quality (blocky/pixelated), the most likely cause is expired YouTube cookies — see the Cookies section below.

8. YouTube Cookies (Important)

YouTube requires authentication cookies to serve high-resolution adaptive streams (1080p, 1440p, 4K). Without valid cookies, yt-dlp falls back to format 18 — a combined 360p stream — causing the blocky/artifacted video quality.

⚠️ Cookies expire periodically (weeks to months). When downloads start looking low quality again, it's time to refresh them.

Why the incognito method is required

YouTube actively rotates cookies on open browser tabs as a security measure. If you export cookies from a regular Chrome window where YouTube is open, those cookies will expire within hours or days. The incognito method exports a "frozen" session that isn't being rotated.

Step-by-step: exporting cookies correctly

  1. Install the "Get cookies.txt LOCALLY" extension in Chrome.
    Important: make sure it's the "LOCALLY" version. The non-LOCALLY version was reported as malware and removed from the Chrome Web Store.
  2. Open a new incognito window in Chrome.
  3. Log into YouTube in that incognito window.
  4. In the same tab, navigate to: https://www.youtube.com/robots.txt
  5. Click the "Get cookies.txt LOCALLY" extension icon and export cookies for youtube.com. Save the file as cookies.txt.
  6. Close the incognito window immediately — this ensures the session is never rotated by an open tab.

🚫 Do NOT use yt-dlp --cookies-from-browser chrome to export cookies. This exports your regular browser cookies, not the frozen incognito session, and they will expire quickly.

Transferring cookies to the server

# Copy from Mac to server
scp ~/Downloads/cookies.txt cyoungers@192.168.0.166:/tmp/cookies.txt

# Install into the Docker container
ssh cyoungers@192.168.0.166 "docker cp /tmp/cookies.txt ytdl-manager:/data/cookies.txt"

Verifying cookies are working

ssh cyoungers@192.168.0.166 "docker exec ytdl-manager yt-dlp \
  --js-runtimes node \
  --cookies /data/cookies.txt \
  --extractor-args 'youtube:player_client=mweb' \
  -F 'https://www.youtube.com/watch?v=jNQXAC9IVRw' \
  2>&1 | grep -v 'sb[0-9]\|storyboard\|mhtml'"

You should see 20+ format options including 1080p and 1440p. If you only see format 18 (360p), the cookies are not valid.

9. Video Filters

All downloads are subject to a built-in --match-filter that silently skips videos that don't meet these criteria:

  • Duration > 3 minutes — skips Shorts and very short clips.
  • Not currently live — skips active live streams.
  • Was not a live stream — skips VODs of past live streams.
  • URL does not contain /shorts/ — belt-and-suspenders Shorts filter.

Additionally, the following error conditions are treated as non-failures and do not mark a subscription as errored:

  • "does not pass filter" — video was skipped by the above rules.
  • "This live event will begin in a few moments" — a scheduled stream not yet started.
  • "Premieres in" — a YouTube Premiere not yet released.

These cases appear in the log but result in Exit code: 0 (success).

10. Reading the Log Viewer

Click any subscription row on the dashboard to open its log. The log file is the raw yt-dlp output for that subscription, appended across all runs. Here's how to read it:

Run header

============================================================
2026-03-22 14:30:00 UTC  [SCHEDULER]
============================================================
Found 3 new video(s)

Each run starts with a timestamp and whether it was triggered by the SCHEDULER or a MANUAL check.

Per-video section

--- Downloading abc123xyz ---
[youtube] abc123xyz: Downloading webpage
[youtube] [jsc:node] Solving JS challenges using node
[info] abc123xyz: Downloading 1 format(s): 399+251
[download] Destination: /downloads/channel/Title_(2026_03_22)_[abc123xyz].f399.mp4
[download] 100% of 850MiB
[Merger] Merging formats into "/downloads/channel/Title_(2026_03_22)_[abc123xyz].mp4"

The format IDs tell you the quality being downloaded:

  • 399+251 — AV1 video (1080p) + Opus audio ✅ Good quality
  • 400+251 — AV1 video (1440p) + Opus audio ✅ Great quality
  • 137+140 — H.264 video (1080p) + AAC audio ✅ Good quality
  • 18 (single number, no +) — combined 360p stream ❌ Cookies are expired

Run footer

Run complete. Exit code: 0
  • Exit code: 0 — success (including filter-skipped videos).
  • Exit code: 1 — one or more videos had a genuine download error. Scroll up to find the ERROR: line.

11. Troubleshooting

Videos look blocky / low quality (360p)

This is almost always expired YouTube cookies. The log will show [info] abc123: Downloading 1 format(s): 18 (a single format number, no +). Fix: refresh cookies using the incognito method described in Section 5.

Exit code: 1 on every run

Open the log and search for ERROR: lines. Common causes:

  • "Sign in to confirm you're not a bot" — cookies are expired or missing.
  • "Video unavailable" — the video was deleted or made private.
  • "Rate limited by YouTube" — too many requests in a short time. Wait an hour and it resolves on its own.
  • "Requested format is not available" — usually also a cookies issue; YouTube is withholding high-res formats.

n challenge solving failed

This warning means yt-dlp's node EJS challenge solver isn't available. Two requirements must both be met: Node.js 22.0.0 or later (the Debian apt nodejs package lags at 20.x and is reported "unsupported" even though node --version works) and the yt-dlp-ejs package, which only comes bundled when yt-dlp is installed with pip install "yt-dlp[default]" rather than plain pip install yt-dlp. Check with docker exec ytdl-manager node --version and docker exec ytdl-manager pip show yt-dlp-ejs.

Subscription stuck in "downloading" status

If a subscription shows as downloading but the log isn't updating, the download process may have hung. Restart the container:

ssh cyoungers@192.168.0.166 "docker restart ytdl-manager"

All subscriptions show "never checked" after a restart

The last_checked times are stored in the SQLite database, which lives in a Docker named volume (ytdl-data). They persist across restarts. If they appear reset, the volume may have been accidentally deleted during a docker compose down -v.

12. Maintenance & Updates

Updating yt-dlp

The easiest way is via the dashboard — if a newer version is available on GitHub, an Update available button appears next to the version number in the header. Click it to update in place without any Docker rebuild.

Alternatively, from the command line:

ssh cyoungers@192.168.0.166 "docker exec ytdl-manager yt-dlp -U"

Deploying Python code changes (no rebuild)

For changes to main.py only (no new dependencies):

ssh cyoungers@192.168.0.166 "cd ~/ai/ytdl-manager-v2 && git pull && \
  docker cp app/main.py ytdl-manager:/app/main.py && \
  docker restart ytdl-manager"

Full Docker rebuild (new dependencies or Dockerfile changes)

ssh cyoungers@192.168.0.166 "cd ~/ai/ytdl-manager-v2 && git pull && \
  docker compose down && \
  docker compose build --no-cache && \
  docker compose up -d"

⚠️ A full rebuild takes 2–5 minutes and downloads ~200MB of packages. Avoid it unless you've changed Dockerfile or requirements.txt.

Accessing the database

ssh cyoungers@192.168.0.166
docker exec -it ytdl-manager python3 -c "
import sqlite3, json
c = sqlite3.connect('/data/subscriptions.db')
c.row_factory = sqlite3.Row
rows = c.execute('SELECT id, name, status, last_checked_at FROM subscriptions').fetchall()
for r in rows: print(dict(r))
"

Re-staggering the schedule

If subscriptions start bunching up and running at the same time, hit the stagger endpoint to redistribute them evenly across their interval windows:

curl -X POST http://192.168.0.166:8912/api/stagger

This reschedules all enabled subscriptions in APScheduler directly — subscriptions with the same interval are spread evenly across that interval. For example, 11 hourly subs will fire ~5.5 minutes apart; 27 six-hourly subs will fire ~13 minutes apart. The staggering persists indefinitely since each subscription recurs independently from its own start time.

Viewing recent successful downloads

ssh cyoungers@192.168.0.166 "docker exec ytdl-manager tail -50 /data/downloads.log"

13. Architecture Reference

Infrastructure

  • Host: Ubuntu, hostname cyoungers-MINI-S (192.168.0.166)
  • Container: ytdl-manager — Python 3.12 + FastAPI + yt-dlp
  • Port: 8912 on host → 8080 in container
  • Repo: https://github.com/cyoungers/ytdl-manager (main)
  • Project path on host: ~/ai/ytdl-manager-v2

Data volume (ytdl-data)

  • /data/subscriptions.db — SQLite database (subscriptions, jobs, download history)
  • /data/archives/{id}.txt — per-subscription yt-dlp download archive (prevents re-downloads)
  • /data/logs/{id}.log — per-subscription yt-dlp output logs
  • /data/cookies.txt — YouTube session cookies
  • /data/downloads.log — plain text log of all successful downloads

Downloads folder

Videos are stored on the NAS at smb://Chris@CJYNAS._smb._tcp.local/video/ytdl-manager — clicking this link opens the folder in Finder.

How a channel check works

  1. Scheduler triggers a check for a subscription based on interval_hours.
  2. For channels: fetch YouTube RSS feed → extract up to 15 video IDs.
  3. For playlists: run yt-dlp --flat-playlist --print id → extract all video IDs.
  4. Compare against the download archive to find new IDs.
  5. For each new video ID: run yt-dlp with format selector, filters, cookies, and n-challenge solver.
  6. On success: append video ID to archive; log the filename to downloads.log.
  7. Wait 5 seconds between videos to avoid YouTube rate limiting.

YouTube client used

Downloads use the mweb (mobile web) player client. This client supports cookies, receives the full list of available formats, and works with yt-dlp's Node.js-based n-challenge solver (bgutil). The android_vr client was previously used but is skipped by yt-dlp when cookies are present.
